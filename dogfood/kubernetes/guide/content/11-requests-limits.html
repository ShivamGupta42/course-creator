Requests and limits

How much CPU and memory you reserve for a container, and what the kernel does when it asks for more.

Picture the idea

Your staging Node web container idles at 80m CPU and 120Mi memory but spikes to 600m during the 40-to-400 request surge; you set a request of 250m and a limit of 700m so the scheduler reserves a realistic floor and the spike still has room.

A request is the table you reserved at a restaurant and a limit is the size of the table: the host seats the room by reservations, and once you sit you cannot spread past your own table even if the place is half empty.

Drop the memory limit on the web Pod from 512Mi to 64Mi and keep the request at 128Mi, then predict whether the Pod gets throttled, OOM-killed, or refused at scheduling time, and which of those leaves a log you can read afterward.

Core ideas

A container declares two numbers per resource. The request is what the scheduler reserves on a node; it is a guarantee and the basis for where the Pod can land. The limit is the ceiling the kernel enforces while the container runs. They answer different questions: the request answers "where does this fit", the limit answers "what happens when it asks for more".

The two resources behave differently at the ceiling. CPU is compressible: a container over its CPU limit is throttled, it runs slower but keeps living. Memory is not compressible: a container over its memory limit is OOM-killed by the kernel and the Pod restarts. That asymmetry is the whole reason a too-low memory limit is dangerous and a too-low CPU limit is merely slow.

Requests also set the QoS class, which decides eviction order when a node runs out of memory. Guaranteed means request equals limit for every resource. Burstable means requests are set but below limits. BestEffort means nothing is set, and those Pods are evicted first.

Your web Pod requests 200m CPU and limits to 400m. The node it runs on is fully booked by requests but mostly idle in real usage. The Pod tries to use 700m during a spike. What happens?

  • It is throttled to 400m even though the node has idle cycles, because the limit is a hard per-container ceiling.
  • It bursts to 700m because the node is idle and limits only apply under contention.
  • It is OOM-killed for exceeding its allocation.
  • The scheduler moves it to a less busy node.

The CPU limit is enforced per container regardless of node idle capacity, so it caps at 400m. The tempting wrong answer treats limits like requests (which only bite under contention); limits bite always. OOM-kill is memory, not CPU, and the scheduler never reschedules a running Pod for load.

A teammate sets the Postgres container's memory request to 256Mi and its limit to 256Mi to "keep it predictable". Under a heavy query it needs 300Mi. What is the operational result?

  • The container is OOM-killed and restarts, because a Guaranteed memory ceiling at 256Mi cannot stretch.
  • It borrows memory from another Pod on the node.
  • It is throttled and the query just runs slower.
  • The scheduler grants 44Mi more from node headroom.

Memory is incompressible, so crossing the limit means OOM-kill, not slowdown. The "runs slower" distractor is the CPU behavior leaking into the memory case, which is the exact confusion this module fixes. Nothing borrows or tops up memory at runtime.

Why does the scheduler bin-pack on requests rather than on what containers actually use?

  • Real usage swings moment to moment, so a guarantee the scheduler can reserve once and trust must be a fixed declared number, not a live reading.
  • Because measuring real usage is impossible in Kubernetes.
  • Because limits would be ignored otherwise.
  • Because requests are always equal to usage.

The scheduler needs a stable contract to make a one-time placement decision; live usage is too volatile to reserve against. Usage is measurable (metrics-server reads it), so the "impossible" option is wrong, and requests rarely equal usage, which is exactly why headroom planning is a separate skill.

Where this shows up

The night your marketing push goes live and traffic climbs from 40 to 400 req/s, the web Pods that worked fine all week start showing OOMKilled in kubectl get pods: the per-request memory crept past a 256Mi limit nobody load-tested, and every restart drops in-flight requests right when you need them most.

The restaurant-table metaphor breaks for CPU bursting. A diner cannot use a neighbor's empty table, but a Burstable Pod genuinely can use idle CPU above its request up to its limit, so CPU requests are a floor you can exceed while a table reservation is not.

Figure it out from scratch

Start With a Real Question

How much CPU and memory should this container reserve?

  • guessing wastes nodes or crashes Pods
  • you need a floor and a ceiling, not one number

Say It in Plain Words

Request is the reservation; limit is the cap.

  • request drives where it lands
  • limit drives what breaks under pressure

What It Is Made Of

Two resources, two ceiling behaviors.

  • CPU: compressible, throttled
  • memory: incompressible, OOM-killed

Now Write the Equation

QoS follows from the gap: request == limit ⇒ Guaranteed

  • set but unequal ⇒ Burstable
  • unset ⇒ BestEffort, evicted first

Try the Simplest Case

One 1000m node, three Pods requesting 300m each.

  • 900m reserved, 100m free
  • a fourth 300m Pod stays Pending

Find Where It Breaks

Requests far below real usage overcommit the node.

  • node looks empty, runs hot
  • memory pressure evicts BestEffort first

Explain It to a Friend

A container asks for two numbers per resource. The request is what the scheduler reserves so the Pod has a guaranteed floor, and it decides which node has room. The limit is the ceiling the kernel enforces once the container runs. CPU over the limit gets throttled and just slows down, but memory over the limit gets the container OOM-killed and restarted. Set both honestly, because under-requesting overcommits the node and a too-tight memory limit kills you during the exact spike you sized everything for.

How to reason about this

Check your understanding

  • Given a Pod that shows OOMKilled, which number do you change, and would raising the request alone fix it?
  • Why can two Pods with identical limits behave differently under node memory pressure?
  • A node has 2000m allocatable CPU and 4 Pods each requesting 600m. Will a fifth 600m Pod schedule, and what is its phase if not?

Show you understand

Take your real web container, read its actual CPU and memory from kubectl top pod over a load test, then set a request at the steady median and a limit above the observed peak, and justify in two sentences why those two numbers, not one, protect both the node and the Pod.

Build it yourself

Write a Deployment manifest with deliberate requests and limits for your web tier, apply it, and observe the QoS class and a forced OOM-kill the cluster reports back.

  1. Worked: gather real numbers. Run kubectl top pod on your current web Pod under light load; note CPU (say 80m) and memory (say 120Mi). The model manifest sets requests: {cpu: 100m, memory: 128Mi}, limits: {cpu: 500m, memory: 256Mi}.
  2. Faded: classify the QoS. Given that manifest, name the QoS class and explain it in one line. Hidden solution: requests are set but below limits, so it is Burstable; check with kubectl get pod <name> -o jsonpath='{.status.qosClass}'.
  3. Independent: force the failure. Lower the memory limit to 64Mi, re-apply, and drive load until it crosses 64Mi. Predict the symptom first, then read it.
  4. Transfer: sanity-check against a node. Your nodes have 2 vCPU (2000m) allocatable. If every web Pod requests 250m, how many fit before the scheduler refuses the next one, and what phase does that next Pod sit in?

You have it when kubectl get pods shows the Pod cycling through OOMKilled after you tightened the memory limit, kubectl describe pod names Reason: OOMKilled in the last state, and you can state the QoS class before checking it. (Verification mode: observed cluster state + an estimate, not a unit-test assert.)

Push further: add a second container to the Pod and predict how its request changes the Pod's total reservation and whether the Pod still fits the node you sized for one container.

Module check

Two questions that decide whether you can size a container under real load.

You raise a Pod's memory request from 128Mi to 256Mi but leave the limit at 256Mi. It was getting OOM-killed at 260Mi. Does the change fix the crashes?

  • No. The crash is the limit, not the request; raising the request only changes scheduling and QoS, so it still dies at 256Mi.
  • Yes, a higher request gives the container more memory to use.
  • Yes, because request and limit are now equal.
  • No, because requests have no effect on anything.

The OOM ceiling is the limit, so only raising the limit above the real peak stops the kills. The tempting answer treats the request as an allocation the container can consume; it is a scheduling reservation, not a runtime grant. Requests do matter (scheduling and eviction order), so the last option is also wrong.

A node shows 90% of its CPU requested but its real CPU usage sits at 20%. A new Pod requesting 300m stays Pending even though the node is mostly idle. Why?

  • Scheduling reserves against requests, not usage, so the node has no unreserved CPU left to fit the new request even though it is idle.
  • The node is broken and needs draining.
  • The new Pod's image failed to pull.
  • Real usage is too low for the scheduler to act.

A Pending Pod is almost always unschedulable, and here the requests are booked even though usage is low, which is the classic overcommit-by-request gap. ImagePull failures show ImagePullBackOff, not Pending, and a healthy idle node is not "broken".
